操作系统-第2章进程管理

什么是进程

进程的组成

同时挂三个QQ号，会对应三个QQ进程，它们的PCB、数据段各不相同，但程序段的内容都是相同的（都是运行着相同的QQ程序）

PCB（进程控制块）：给操作系统用的

- 进程描述信息

- 进程控制和管理信息

- 资源分配清单

- 处理机相关信息

程序段、数据段：给进程自己用的，与进程自身的运行逻辑有关

- 程序段：程序的代码（指令序列）

- 数据段：运行过程中产生的各种数据（如：程序中定义的变量）

更确切的说，应该是“进程实体（进程映像）的组成”

程序段、数据段、PCB三部分组成了进程实体（进程映像）

引入进程实体的概念后，可把进程定义为：
进程是进程实体的运行过程，是系统进行资源分配和调度的一个独立单位。

注意：PCB是进程存在的唯一标志！

注：一个进程被“调度”，就是指操作系统决定让这个进程上CPU运行。

进程间的转换

进程状态转换流程

- 创建态 → 就绪态：系统完成创建进程的一系列工作

- 就绪态 → 运行态：进程被调度（操作系统分配处理机）

- 运行态 → 就绪态：分给进程的时间片用完、有更高优先级的进程进入就绪队列

- 运行态 → 阻塞态：进程用“系统调用”的方式申请某种系统资源，或者请求等待某个事件发生（主动行为，进程自身控制）

- 阻塞态 → 就绪态：进程申请的资源被分配，或等待的事件发生（被动行为，非进程自身控制）

- 运行态 → 终止态：进程运行结束，或运行过程中遇到不可修复的错误

注意：不能由阻塞态直接转换为运行态，也不能由就绪态直接转换为阻塞态（因为进入阻塞态是进程主动请求的，必然需要进程在运行时才能发出这种请求）

进程控制

进程控制就是要实现进程状态的转换，由原语实现。

基本概念

- 原语：一种特殊的程序，执行必须一气呵成，不可中断，用关/开中断来实现。

相关原语

- 进程的创建

- 进程的终止

- 进程的阻塞

- 进程的唤醒（阻塞和唤醒要成对出现）

- 进程的切换

无论哪个进程控制原语，要做的无非三类事情：

1. 更新PCB中的信息（修改进程状态state、保存/恢复运行环境）

2. 将PCB插入合适的队列

3. 分配/回收资源

进程间通信IPC

消息传递（直接通信方式）

- 发送原语：send(Q, msg)

- 接收原语：receive(P, &msg)

- 消息头包括：发送进程ID、接受进程ID、消息长度等格式化的信息

特点：点名道姓的消息传递，进程直接通信。

消息传递（间接通信方式）

- 发送原语：send(A, msg)（往信箱A发送消息msg）

- 接收原语：receive(A, &msg)（从信箱A接受消息）

特点：以信箱作为中间实体进行消息传递，可以多个进程往同一个信箱send消息，也可以多个进程从同一个信箱中receive消息。

管道通信

“管道”是一个特殊的共享文件，又名pipe文件，其实就是在内存中开辟一个大小固定的内存缓冲区。

核心规则（5点）：

1. 管道只能采用半双工通信，某一时间段内只能实现单向的传输。如果要实现双向同时通信，则需要设置两个管道。

2. 各进程要互斥地访问管道（由操作系统实现）。

3. 当管道写满时，写进程将阻塞，直到读进程将管道中的数据取走，即可唤醒写进程。

4. 当管道读空时，读进程将阻塞，直到写进程往管道中写入数据，即可唤醒读进程。

5. 管道中的数据一旦被读出，就彻底消失。因此，当多个进程读同一个管道时，可能会错乱。对此，通常有两种解决方案：
① 一个管道允许多个写进程，一个读进程（2014年408真题高教社官方答案）；
② 允许有多个写进程，多个读进程，但系统会让各个读进程轮流从管道中读数据（Linux的方案）。

信号通信

实现原理——发送与保存

信号用于通知进程某个特定事件已经发生，可由用户进程之间发送，也可由内核进程给用户进程发送。

- 常用发送信号的函数：int kill(pid_t pid, int sig)（参数：pid=发给谁，sig=发什么信号）

- PCB中用两个Nbit位向量保存信号：
- 待处理信号pending：标记哪些信号已收到但未处理
- 被阻塞信号blocked（信号掩码signal mask）：标记哪些信号被屏蔽，暂时不处理

示例：进程P1的pending=10000001（表示信号1和8待处理），blocked=00000011（表示信号7和8被阻塞），则仅处理信号1。

实现原理——信号的处理（When）

处理时机：当进程从内核态转为用户态时（如：系统调用返回、或中断处理返回时），例行检查是否有待处理信号，如果有，就处理信号。

处理流程：

1. 进程收到信号

2. 从内核态回到用户态时，检测到待处理信号（排除被阻塞的信号）

3. 转移到对应信号的处理程序

4. 处理完成后，返回到原进程的下一条指令继续执行

知识回顾与重要考点

- 作用：实现简单的进程间通信；常作为异常处理的一种配套机制（拓展知识）。

- 信号的发送：可由一个进程给另一个进程发送（kill函数），也可由进程给自己发送。

- 信号的保存：PCB中的两个Nbit位向量（pending、blocked）。

- 处理时机：进程从内核态转为用户态时。

- 信号的处理方式：
① 执行默认的信号处理程序（部分信号默认忽略，如Linux的SIGWINCH信号）；
② 执行用户定义的信号处理程序（覆盖默认处理）。

- 注意点：
- 不同操作系统，信号类型定义不同；
- 重复收到的同类信号，会被简单丢弃（仅用1bit记录）；
- 有些信号既不能自定义处理，也不能阻塞（如Linux的SIGKILL、SIGSTOP信号）。

线程

线程的实现方式

- 用户级线程：从用户视角能看到的线程，由线程库实现。

- 内核级线程：从操作系统视角看到的线程，由操作系统实现（内核级线程才是处理机分配的单位）。

- 组合方式：上述两种方式的结合。

多线程模型

1. 一对一模型：一个用户级线程映射到一个内核级线程
- 优点：各个线程可分配到多核处理机并行执行，并发度高；
- 缺点：线程管理需要操作系统支持，开销大。

2. 多对一模型：多个用户级线程映射到一个内核级线程
- 优点：线程管理开销小，效率高；
- 缺点：一个线程阻塞会导致整个进程都被阻塞（并发度低）。

3. 多对多模型：n个用户级线程映射到m个内核级线程（n≥m）
- 优点：集一对一和多对一模型之所长。

TCB（线程控制块）

线程表（thread table）：可将多个TCB组织成一张线程表，线程切换时需保存/恢复TCB中的信息。

- 线程标识符（TID）：与PID类似。

- 程序计数器（PC）：线程目前执行到哪里。

- 其他寄存器：线程运行的中间结果。

- 堆栈指针：堆栈保存函数调用信息、局部变量等。

- 线程运行状态：运行/就绪/阻塞。

- 优先级：线程调度、资源分配的参考。

补充知识

七状态模型：在五状态模型的基础上加入了“就绪挂起”和“阻塞挂起”两种状态。

进程调度的时机与禁止时机

需要进行进程调度与切换的情况（进程放弃处理机）

- 主动放弃：进程正常终止、运行过程中发生异常而终止、进程主动请求阻塞（如等待I/O）。

- 被动放弃：分给进程的时间片用完、有更紧急的事需要处理（如I/O中断）、有更高优先级的进程进入就绪队列。

注：有的系统只允许进程主动放弃处理机；有的系统允许主动放弃，也可强行剥夺（被动放弃）；进程在普通临界区中可进行调度、切换。

不能进行进程调度与切换的情况

1. 在处理中断的过程中（部分系统可优化实现，但通常禁止）。

2. 进程在操作系统内核程序临界区中。

3. 在原子操作过程中（原语）：原子操作不可中断，需一气呵成（如修改PCB中程序状态标志，并把PCB放到相应队列）。

进程调度的方式

1. 非剥夺调度方式（非抢占方式）
- 规则：只允许进程主动放弃处理机，即使有更紧迫的任务到达，当前进程仍继续执行，直到终止或主动阻塞。
- 优点：实现简单，系统开销小；
- 缺点：无法及时处理紧急任务，适合早期批处理系统。

2. 剥夺调度方式（抢占方式）
- 规则：当进程正在执行时，若有更重要/紧迫的进程需要处理机，立即暂停当前进程，将处理机分配给该进程。
- 优点：可优先处理紧急进程，可实现时间片轮流执行（时钟中断），适合分时操作系统、实时操作系统；
- 缺点：系统开销较大（频繁切换）。

补充要点

- 狭义“调度”与“切换”的区别：调度是选择进程，切换是保存原进程数据、恢复新进程数据。

- 重要结论：进程调度、切换是有代价的，并不是调度越频繁，并发度就越高。

调度算法评价

核心评价指标：公平性、平均周转时间、平均等待时间、响应时间、吞吐量等。

（注：文档中此处重复了“进程调度时机与方式”的知识点，已去重，保留核心评价逻辑）

调度算法

1. FCFS 先来先服务

- 规则：按进程到达就绪队列的先后顺序调度，非抢占式。

- 优点：公平、实现简单。

- 缺点：对短作业不利，平均等待时间长。

- 是否饥饿：不会。

- 是否考虑等待时间/运行时间：仅考虑等待时间（先后顺序），不考虑运行时间。

2. SJF 短作业优先

核心：优先调度运行时间最短的进程，分为非抢占式和抢占式（最短剩余时间优先SRTN）。

- 非抢占式：仅在当前进程主动放弃处理机时，选择剩余运行时间最短的就绪进程。

- 抢占式（SRTN）：每当有进程加入就绪队列，就重新调度；若新进程剩余时间比当前运行进程短，则抢占处理机。

- 优点：“最短的”平均等待时间、平均周转时间。

- 缺点：对长作业不利，可能导致长作业饥饿；难以做到真正的“短作业”（无法预知进程运行时间）。

- 是否饥饿：会。

- 是否考虑等待时间/运行时间：仅考虑运行时间。

计算结果（SRTN）：

- 平均周转时间=(16+5+1+6)/4=7；平均带权周转时间≈(2.28+1.25+1+1.5)/4=1.50；平均等待时间=(9+1+0+2)/4=3

3. 高响应比优先 HRRN

核心：非抢占式，仅在当前进程主动放弃处理机时调度，选择响应比最高的就绪进程。

响应比公式：响应比 = （等待时间 + 要求服务时间） / 要求服务时间

- 优点：折中FCFS和SJF，综合考虑等待时间和运行时间，无饥饿。

- 缺点：需要计算响应比，开销略大。

- 是否饥饿：不会。

- 是否考虑等待时间/运行时间：两者都考虑。

4. 时间片轮转调度 RR

核心：抢占式，轮流让就绪队列中的进程依次执行一个时间片（选择队头进程），重点关注响应时间。

- 优点：公平，适用于分时系统（如桌面操作系统），响应时间快。

- 缺点：频繁切换有开销；时间片大小影响性能。

- 时间片大小注意：
- 太大：退化为FCFS，响应时间增大；
- 太小：切换频繁，系统开销大，实际执行时间占比降低。

- 是否饥饿：不会。

RR例题

进程信息（同前），时间片分别为2、5时：

- 时间片=2：进程执行顺序为P1→P2→P1→P3→P4→P2→P1→P4，响应时间快，但切换频繁。

- 时间片=5：P1执行5个时间片后，剩余2个时间片；P2到达后等待，P1完成后P2执行，接近FCFS，响应时间增大。

5. 优先级调度

核心：按进程优先级调度，可分为抢占式（高优先级进程抢占低优先级进程）和非抢占式（仅当前进程放弃后调度）。

- 优先级设置原则（通常）：
- 系统进程优先级高于用户进程；
- 前台进程优先级高于后台进程；
- 操作系统更偏好I/O型进程（I/O繁忙型），而非计算型进程（CPU繁忙型）。

- 优点：可优先处理紧急进程，适用于实时系统。

- 缺点：可能导致低优先级进程饥饿。

- 是否饥饿：会。

6. 多级反馈队列

核心：抢占式，设置多个优先级队列，进程按优先级进入对应队列，优先级越高，时间片越小；进程在低优先级队列中等待时间过长，可提升优先级（避免饥饿）。

- 优点：平衡响应时间和吞吐量，兼顾短作业和长作业，无明显缺点。

- 缺点：实现复杂，可能导致低优先级进程饥饿（需优化优先级提升机制）。

- 是否饥饿：会（未优化时）。

7. 多级队列优先级算法

核心：按进程类型设置多个队列（如交互式进程、批处理进程），进程创建后插入对应队列，队列间采用固定优先级或时间片划分，各队列内部采用不同调度策略。

- 队列间策略：
- 固定优先级：高优先级队列空时，低优先级队列才能调度；
- 时间片划分：如三个队列分配50%、40%、10%的时间片。

- 队列内策略：系统进程队列用优先级调度，交互式队列用RR，批处理队列用FCFS。

8. 多处理机调度

与单处理机调度的区别：不仅要决定哪个就绪进程优先上处理机，还要决定该进程上哪个CPU。

常用调度算法：FCFS、SJF、RR、优先级调度等（与单处理机一致）。

核心目标：负载均衡、处理机亲和性

1. 公共就绪队列
- 特点：所有CPU共享一个就绪队列，每个CPU运行调度程序时从队列中选进程（需上锁保证互斥）。
- 优点：天然实现负载均衡；
- 缺点：进程频繁换CPU，处理机亲和性差（Cache失效）。

2. 私有就绪队列
- 特点：每个CPU有自己的就绪队列，CPU空闲时从自身队列选进程。
- 优点：天然实现处理机亲和性；
- 缺点：易出现负载不均衡，需通过“拉迁移”（低负载CPU从高负载CPU拉进程）或“推迁移”（高负载CPU往低负载CPU推进程）实现负载均衡。

处理机亲和性提升方法

- 软亲和：由进程调度程序尽量保证进程在同一个CPU上运行。

- 硬亲和：由用户进程通过系统调用，主动要求操作系统分配固定的CPU。

同步与互斥

1. 进程同步

知识点回顾：进程具有异步性（各并发进程以独立、不可预知的速度推进）。

定义：同步亦称直接制约关系，是指为完成某种任务而建立的两个或多个进程，因需要在某些位置协调工作次序而产生的制约关系（源于进程间的相互合作）。

示例：管道通信中，写进程“写数据”和读进程“读数据”必须按“写→读”的顺序执行，这就是同步关系。

2. 进程互斥

前提：进程的“并发”需要“共享”支持，各进程不可避免地需要共享系统资源（内存、打印机等）。

资源共享方式

- 互斥共享方式：一个时间段内只允许一个进程访问该资源（如打印机、摄像头）。

- 同时共享方式：一个时间段内允许多个进程“同时”访问该资源（如内存）。

临界资源：一个时间段内只允许一个进程使用的资源（物理设备、变量、数据、内存缓冲区等）。

互斥定义：亦称间接制约关系，当一个进程访问某临界资源时，另一个想要访问该资源的进程必须等待，直到当前进程访问结束并释放资源。

3. 临界区相关

临界区四部分

1. 进入区：检查是否可进入临界区，若可进入，需要“上锁”。

2. 临界区：访问临界资源的那段代码。

3. 退出区：负责“解锁”。

4. 剩余区：其余代码部分。

临界区访问原则（4点）

1. 空闲让进：临界区空闲时，应允许一个进程访问。

2. 忙则等待：临界区正在被访问时，其他试图访问的进程需要等待。

3. 有限等待：要在有限时间内进入临界区，保证不会饥饿。

4. 让权等待：进不了临界区的进程，要释放处理机，防止忙等。

进程互斥的实现方法

一、软件实现方法

1. 单标志法

核心：用turn变量表示当前允许进入临界区的进程号（如turn=0表示允许P0进入），进程访问完后修改turn值，实现“谦让”。

int turn=0; // turn表示当前允许进入临界区的进程号

// P0进程
while (turn!=0); // 进入区：检查是否轮到自己
critical section; // 临界区
turn =1; // 退出区：谦让，让P1进入
remainder section; // 剩余区

// P1进程
while (turn !=1); // 进入区
critical section; // 临界区
turn=0; // 退出区
remainder section; // 剩余区

问题：违背“空闲让进”原则。若允许进入的进程（如P0）不访问临界区，即使临界区空闲，其他进程（如P1）也无法访问。

2. 双标志先检查法

核心：用布尔型数组flag表示进程进入临界区的意愿（flag[i]=true表示进程i想进入），进程先检查其他进程是否想进入，若无则标记自身意愿，再进入临界区。

bool flag[2];
flag[0] = false; flag[1] = false; // 初始：均不想进入

// P0进程
while (flag[1]); // 进入区：检查P1是否想进入
flag[0]=true; // 标记自身意愿
critical section; // 临界区
flag[0]=false; // 退出区：取消标记
remainder section; // 剩余区

// P1进程
while (flag[0]); // 进入区
flag[1]=true; // 标记自身意愿
critical section; // 临界区
flag[1]=false; // 退出区
remainder section; // 剩余区

问题：违背“忙则等待”原则。进入区的“检查”和“上锁”（标记flag）不是原子操作，可能出现两个进程同时进入临界区（如P0检查flag[1]=false后，切换到P1，P1检查flag[0]=false，两者均标记flag并进入临界区）。

3. 双标志后检查法

核心：改进双标志先检查法，先“上锁”（标记flag），再检查其他进程是否想进入，避免同时进入。

bool flag[2];
flag[0] = false; flag[1] = false;

// P0进程
flag[0]=true; // 先标记意愿（上锁）
while (flag[1]); // 再检查P1是否想进入（进入区）
critical section; // 临界区
flag[0]=false; // 退出区
remainder section; // 剩余区

// P1进程
flag[1]=true; // 先标记意愿
while (flag[0]); // 再检查P0是否想进入
critical section; // 临界区
flag[1]=false; // 退出区
remainder section; // 剩余区

问题：违背“空闲让进”和“有限等待”原则，可能导致饥饿。若P0和P1同时标记flag为true，两者都会陷入循环等待，无法进入临界区。

4. Peterson算法

核心：结合双标志法和单标志法，进程先标记进入意愿，再“谦让”（修改turn值），最后检查对方意愿和自身谦让状态，实现互斥。

bool flag[2] = {false, false}; // 进入意愿，初始均为false
int turn=0; // 优先让哪个进程进入

// P0进程
flag[0]=true; // 标记进入意愿
turn=1; // 谦让，优先让P1进入
while (flag[1] && turn==1); // 进入区：P1想进入且最后是自己谦让，等待
critical section; // 临界区
flag[0]=false; // 退出区
remainder section; // 剩余区

// P1进程
flag[1]=true; // 标记进入意愿
turn=0; // 谦让，优先让P0进入
while(flag[0] && turn==0); // 进入区：P0想进入且最后是自己谦让，等待
critical section; // 临界区
flag[1] = false; // 退出区
remainder section; // 剩余区

特点：遵循空闲让进、忙则等待、有限等待三个原则，但未遵循让权等待原则（等待时会忙等）。

二、硬件实现方法

1. 利用开/关中断指令实现

核心：与原语实现思想一致，进程进入临界区前关中断，禁止进程切换；访问完后开中断，允许切换。

关中断; // 禁止中断，避免进程切换
critical section; // 临界区
开中断; // 允许中断，恢复进程切换
remainder section; // 剩余区

- 优点：简单、高效。

- 缺点：不适用于多处理机；只适用于操作系统内核进程，不适用于用户进程（开/关中断指令只能运行在内核态，用户使用危险）。

2. TestAndSet（TS/TSL）指令

核心：硬件实现的原子操作（不可中断），将“检查”和“上锁”一气呵成，用Lock变量标记临界区是否被加锁（true=加锁，false=未加锁）。

// TSL指令逻辑（C语言描述）
bool TestAndSet (bool *Lock){
    bool old;
    old=*Lock; // 保存Lock原来的值
    *Lock=true; // 无论之前是否加锁，都设为加锁
    return old; // 返回原来的值
}

// 用TSL实现互斥
bool Lock = false; // 初始未加锁
while(TestAndSet(&Lock)); // 进入区：上锁并检查，未加锁则跳出循环
critical section; // 临界区
Lock = false; // 退出区：解锁
remainder section; // 剩余区

- 优点：实现简单，无逻辑漏洞；适用于多处理机环境。

- 缺点：不满足“让权等待”原则，等待进程会忙等。

3. Swap（Exchange/XCHG）指令

核心：硬件实现的原子操作，交换两个变量的值，逻辑与TSL类似，用Lock标记临界区状态。

// Swap指令逻辑（C语言描述）
Swap(bool*a, bool *b){
    bool temp;
    temp=*a;
    *a=*b;
    *b=temp;
}

// 用Swap实现互斥
bool Lock = false;
bool old = true;
while (old ==true)
    Swap(&Lock,&old); // 进入区：交换Lock和old，未加锁则old变为false，跳出循环
critical section; // 临界区
Lock = false; // 退出区：解锁
remainder section; // 剩余区

- 优点：实现简单，适用于多处理机环境。

- 缺点：不满足“让权等待”原则，等待进程会忙等。

三、互斥锁

核心：由操作系统提供的互斥机制，进程通过“加锁”（lock）和“解锁”（unlock）操作访问临界区，本质是对硬件实现方法的封装，简化编程。

信号量机制（P、V操作）

基本概念

- 信号量：一个变量（整数或记录型），表示系统中某种资源的数量（如打印机数量为1，信号量初值为1）。

- 原语：特殊程序段，执行一气呵成，不可中断（由关/开中断实现），解决软件实现中“进入区操作无法原子化”的问题。

- P、V操作：wait(S)原语（P操作）和signal(S)原语（V操作）的简称（来自荷兰语proberen和verhogen），用于操作信号量。

1. 整型信号量

核心：用整数变量作为信号量，表示资源数量，仅支持初始化、P操作、V操作三种操作。

// 示例：系统中有1台打印机，信号量S初值为1
int S=1; // 初始化：可用资源数为1

// P操作（wait原语）：申请资源，进入区
void wait(int S){
    while(S<=0); // 资源不足，忙等
    S=S-1; // 占用一个资源
}

// V操作（signal原语）：释放资源，退出区
void signal (int S){
    S=S+1; // 释放一个资源
}

// 进程调用示例
进程P0:
wait(S); // 申请打印机
使用打印机; // 临界区
signal(S); // 释放打印机

进程P1:
wait(S); // 申请打印机
使用打印机; // 临界区
signal(S); // 释放打印机

问题：不满足“让权等待”原则，资源不足时进程会忙等。

2. 记录型信号量

核心：用记录型数据结构表示信号量，包含“剩余资源数”和“等待队列”，解决整型信号量的忙等问题，遵循让权等待原则。

// 记录型信号量定义
typedef struct {
    int value; // 剩余资源数
    struct process* L; // 等待该资源的进程队列
} semaphore;

// P操作（wait原语）：申请资源
void wait (semaphore S){
    S.value--; // 申请一个资源，资源数减1
    if(S.value < 0){ // 资源不足，进程阻塞
        block(S.L); // 进程从运行态→阻塞态，插入等待队列
    }
}

// V操作（signal原语）：释放资源
void signal (semaphore S){
    S.value++; // 释放一个资源，资源数加1
    if(S.value <= 0){ // 仍有进程等待该资源
        wakeup(S.L); // 唤醒等待队列中的第一个进程，阻塞态→就绪态
    }
}

关键理解

- S.value初值：表示系统中该类资源的总数。

- P操作：S.value--后，若S.value<0，说明资源已分配完毕，当前进程阻塞，主动放弃处理机（让权等待）。

- V操作：S.value++后，若S.value≤0，说明仍有进程在等待，唤醒等待队列中的进程。
