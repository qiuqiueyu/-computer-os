操作系统-第5章设备管理

一、I/O设备分类

按信息交换的单位分类

1. 块设备

传输速率较高，可寻址，即对它可随机地读/写任一块；数据传输的基本单位是“块”；示例：磁盘等。

2. 字符设备

传输速率较慢，不可寻址，在输入/输出时常采用中断驱动方式；数据传输的基本单位是字符；示例：鼠标、键盘等。

二、I/O控制器（设备控制器）

CPU无法直接控制I/O设备的机械部件，因此I/O设备需有一个电子部件作为CPU和I/O设备机械部件之间的“中介”，即I/O控制器，又称设备控制器。CPU可控制I/O控制器，再由I/O控制器控制设备的机械部件。

（一）I/O控制器的功能

- 接受和识别CPU发出的命令：I/O控制器中会有相应的控制寄存器来存放命令和参数。

- 向CPU报告设备的状态：I/O控制器中会有相应的状态寄存器，用于记录I/O设备的当前状态（如：1表示空闲，0表示忙碌）。

- 数据交换：I/O控制器中会设置相应的数据寄存器。输出时，数据寄存器用于暂存CPU发来的数据，之后再由控制器传送设备；输入时，数据寄存器用于暂存设备发来的数据，之后CPU从数据寄存器中取走数据。

- 地址识别：由I/O逻辑实现。

（二）I/O控制器的组成

- CPU与控制器的接口：负责接收和识别CPU的各种命令（如地址译码），并负责对设备发出命令；实现CPU与控制器之间的通信，CPU通过控制线发出命令、地址线指明要操作的设备、数据线取出/放入数据。

- I/O逻辑：负责识别CPU发出的命令，并向设备发出命令。

- 控制器与设备的接口：用于实现控制器与设备之间的通信，传送输入/输出数据。

- 核心寄存器：包括数据寄存器（暂存数据）、控制寄存器（存放CPU命令）、状态寄存器（记录设备状态）。

三、寄存器编址

两种编址方式对比

1. 内存映像I/O

控制器中的寄存器与内存地址统一编址；可以采用对内存进行操作的指令来对控制器进行操作。

2. 寄存器独立编址

控制器中的寄存器使用单独的地址；需要设置专门的指令来操作控制器。

四、I/O控制方式

核心问题：用什么样的方式来控制I/O设备的数据读/写；关键关注：1. 完成一次读/写操作的流程；2. CPU干预的频率。

（一）程序直接控制方式（轮询方式）

1. 读/写操作流程

CPU向控制器发出读命令 → 读I/O模块的状态并读入CPU寄存器 → 检查状态，若未准备好则循环轮询，若准备好则从I/O模块中读取字 → 将CPU寄存器中的内容写到主存中 → 重复直至完成所有读写，执行下一条指令（若出现错误则根据错误条件处理）。

2. 核心特点

- CPU干预频率：极高，I/O操作开始之前、完成之后需要CPU介入，且等待I/O完成的过程中CPU需不断轮询检查。

- 数据传输单位：每次读/写一个字。

- 数据流向：读操作（输入）：I/O设备→CPU→内存；写操作（输出）：内存→CPU→I/O设备；每个字的读/写都需要CPU帮助。

- 优缺点：优点是实现简单，在读/写指令后添加循环检查指令即可；缺点是CPU和I/O设备只能串行工作，CPU长期处于“忙等”状态，利用率极低。

（二）中断驱动方式

1. 读/写操作流程

CPU向I/O模块发出读命令后，可去做其他事情 → I/O模块完成准备后，向CPU发送中断信号 → CPU响应中断，读I/O模块的状态 → 检查状态，若有错误则处理，若准备好则从I/O模块中读取字 → 将数据写入存储器 → 重复直至完成所有读写，执行下一条指令。

2. 核心特点

- CPU干预频率：每次I/O操作开始之前、完成之后需要CPU介入，等待I/O完成的过程中CPU可切换到别的进程执行。

- 数据传输单位：每次读/写一个字。

- 数据流向：与程序直接控制方式一致（I/O设备↔CPU↔内存）。

- 优缺点：优点是CPU和I/O设备可并行工作，CPU利用率明显提升；缺点是每个字的传输都需经过CPU，频繁的中断处理会消耗较多CPU时间。

（三）DMA方式（直接内存访问）

1. DMA控制器组成

DR（数据寄存器）：暂存从设备到内存或从内存到设备的数据；MAR（内存地址寄存器）：输入时指明数据在内存的存放位置，输出时指明要输出数据的内存位置；DC（数据计数器）：表示剩余要读/写的字节数；CR（命令/状态寄存器）：存放CPU发来的I/O命令或设备状态信息。

2. 读/写操作流程

CPU向DMA控制器发出读块命令后，可去做其他事情 → DMA控制器自行完成数据块传输 → 传输完成后，DMA控制器向CPU发送中断信号 → CPU响应中断，读取DMA模块的状态，执行后续操作。

3. 核心特点

- CPU干预频率：仅在传送一个或多个数据块的开始和结束时，才需要CPU干预。

- 数据传输单位：每次读/写一个或多个块（需是连续的块，且在内存中也连续）。

- 数据流向：不再经过CPU，读操作：I/O设备→内存；写操作：内存→I/O设备。

- 优缺点：优点是数据传输效率高，CPU和I/O设备并行性提升；缺点是CPU每发出一条I/O指令，只能读/写一个或多个连续的数据块。

（四）通道控制方式

1. 通道简介

通道是一种硬件，可理解为“弱鸡版的CPU”，能识别并执行一系列通道指令；与CPU相比较，通道可执行的指令单一，且通道程序存放在主机内存中，与CPU共享内存。

2. 读/写操作流程

CPU给通道发出指令后，可去做其他事情 → 通道自主执行通道程序，完成I/O操作 → 操作完成后，通道向CPU发送中断信号 → CPU进行中断处理，执行后续操作。

3. 核心特点

- CPU干预频率：极低，仅在发出指令和通道完成操作后需要CPU干预。

- 数据传输单位：每次读/写一组数据块。

- 数据流向：在通道控制下，读操作：I/O设备→内存；写操作：内存→I/O设备。

- 优缺点：优点是CPU、通道、I/O设备可并行工作，资源利用率很高；缺点是实现复杂，需要专门的通道硬件支持。

（五）四种控制方式总结

整体发展趋势：尽量减少CPU对I/O过程的干预，将CPU从繁杂的I/O控制事务中解脱出来，专注于数据处理任务。

五、I/O软件硬件层次

（一）层次结构（从上层到下层）

1. 用户层软件：实现与用户交互的接口，向上提供方便易用的库函数，用户通过库函数、系统调用发起I/O请求。

2. 设备独立性软件（操作系统内核部分）：核心功能包括：建立逻辑设备名到物理设备名的映射关系；根据设备类型选择调用相应的驱动程序；向上层提供统一的调用接口（如read/write系统调用）；设备的保护、差错处理、设备的分配与回收、数据缓冲区管理。

3. 设备驱动程序（操作系统内核部分）：直接与硬件打交道，设置设备寄存器、检查设备状态；不同类型的I/O设备需要不同的驱动程序（因不同设备内部硬件特性不同，厂家需提供对应驱动）。

4. 中断处理程序（操作系统内核部分）：当I/O任务完成时，I/O控制器会发送中断信号，系统根据中断信号类型找到相应的中断处理程序执行，处理流程：从控制器读出设备状态→判断I/O是否正常→若异常则根据原因处理，若正常则从设备中读入一个字的数据并经由CPU放到内存缓冲区中→进行I/O应答（清除中断信号）。

5. 硬件：执行I/O操作，由机械部件和电子部件（I/O控制器）组成。

（二）逻辑设备表（LUT）管理方式

- 方式一：整个系统只设置一张LUT，所有用户不能使用相同的逻辑设备名，适用于单用户操作系统。

- 方式二：为每个用户设置一张LUT，各个用户使用的逻辑设备名可以重复，适用于多用户操作系统；系统在用户登录时为其建立用户管理进程，LUT存放在该进程的PCB中。

（三）关键说明

I/O设备被当做一种特殊的文件；直接涉及硬件具体细节、且与中断无关的操作，在设备驱动程序层完成；不涉及硬件、对各种设备都需要进行的管理工作，在设备独立性软件层完成。

六、输入输出应用程序接口

（一）接口分类

- 字符设备接口：对应字符设备（如键盘、打印机），不可寻址，每次读/写一个字符。

- 块设备接口：对应块设备（如磁盘），可寻址，每次读/写一个块。

- 网络设备接口：对应网络控制器（网卡），负责网络数据的收发。

- 设备驱动程序接口：操作系统规定统一的接口标准，各厂商按要求开发设备驱动程序（如read(p1,p2,p3)接口）。

（二）阻塞/非阻塞I/O

- 阻塞I/O：应用程序发出I/O系统调用后，进程需转为阻塞态等待；示例：从键盘读一个字符（get）。

- 非阻塞I/O：应用程序发出I/O系统调用后，系统调用可迅速返回，进程无需阻塞等待；示例：往磁盘写数据（write）。

（三）关键说明

用户层的应用程序无法用一个统一的系统调用接口完成所有类型设备的I/O，需通过不同的接口对应不同类型的设备。

七、I/O核心子系统

（一）假脱机技术（SPOOLing技术）

1. 脱机技术基础

批处理阶段引入脱机输入/输出技术（用磁带完成）：在外围控制机的控制下，慢速输入设备的数据先被输入到速度更快的磁带上，之后主机从磁带上读入数据，缓解速度矛盾；输出时流程类似。“脱机”即脱离主机的控制进行输入/输出操作。

2. 假脱机技术简介

假脱机技术用软件的方式模拟脱机技术，需多道程序技术支持；系统会建立“输入进程”和“输出进程”，其中输入进程模拟脱机输入时的外围控制机，输出进程模拟脱机输出时的外围控制机。

3. 系统组成

输入进程、输出进程、输入井、输出井（磁盘上的缓冲区）、输入缓冲区、输出缓冲区（内存中的缓冲区）、输入设备、输出设备。

4. 共享打印机原理（假脱机技术应用）

打印机是独占式设备（一段时间内只能满足一个进程的请求），通过假脱机技术可改造成共享设备，流程如下：

1. 当多个用户进程提出打印请求时，系统不直接分配打印机，而是由假脱机管理进程为每个进程做两件事：①在磁盘输出井中申请一个空闲缓冲区，将打印数据送入其中；②让用户进程填写打印请求表（说明打印数据存放位置等信息），并将该表挂到假脱机文件队列（打印任务队列）上。

2. 当打印机空闲时，输出进程从文件队列的队头取出一张打印请求表，根据表中要求将数据从输出井传送到输出缓冲区，再输出到打印机，依次处理所有打印任务。

5. 相关概念

- 独占式设备：只允许各个进程串行使用的设备，示例：未使用假脱机技术的打印机。

- 共享设备：允许多个进程“宏观上同时使用”（微观上交替使用）的设备，可同时满足多个进程的请求。

（二）设备分配管理

1. 设备固有属性

独占设备、共享设备、虚拟设备（通过SPOOLing技术实现）。

2. 设备分配应考虑的因素

- 分配算法：先来先服务、优先级高者优先、短任务优先等。

- 安全性：安全分配方式、不安全分配方式。

- 分配方式：静态分配（进程运行前分配全部所需资源，运行结束后归还）、动态分配（进程运行过程中动态申请设备资源）。

3. 设备分配管理中的数据结构

- 设备控制表（DCT）：每个设备对应一张，关键字段：设备类型、标识符、状态、指向COCT的指针、等待队列指针。

- 控制器控制表（COCT）：每个控制器对应一张，关键字段：状态、指向CHCT的指针、等待队列指针。

- 通道控制表（CHCT）：每个通道对应一张，关键字段：状态、等待队列指针。

- 系统设备表（SDT）：记录整个系统中所有设备的情况，每个设备对应一个表目，关键字段：设备类型、标识符、DCT、驱动程序入口。

4. 设备分配步骤（原始）

1. 根据进程请求的物理设备名查找SDT；

2. 根据SDT找到DCT并分配设备；

3. 根据DCT找到COCT并分配控制器；

4. 根据COCT找到CHCT并分配通道。

注：只有设备、控制器、通道三者都分配成功，此次设备分配才算成功，之后可启动I/O设备进行数据传送。

缺点：用户编程时必须使用物理设备名，换设备则程序无法运行；若请求的物理设备忙碌，即使有同类型空闲设备，进程也需阻塞等待。

5. 设备分配步骤的改进

1. 根据进程请求的逻辑设备名查找SDT（用户编程时提供的逻辑设备名即设备类型）；

2. 查找SDT，找到指定类型的空闲设备，分配给进程，并在LUT中新增一个表项；

3. 根据DCT找到COCT，若控制器忙碌则将进程PCB挂到控制器等待队列，否则分配控制器；

4. 根据COCT找到CHCT，若通道忙碌则将进程PCB挂到通道等待队列，否则分配通道。

6. 逻辑设备表（LUT）的设置问题

- 整个系统一张LUT：各用户所用的逻辑设备名不允许重复。

- 每个用户一张LUT：各个用户的逻辑设备名可重复。

八、缓冲区

（一）缓冲区的作用

- 缓和CPU与I/O设备之间速度不匹配的矛盾。

- 减少对CPU的中断频率，放宽对CPU中断响应时间的限制（如字符设备每输出一个字符需发一次中断，通过缓冲区可批量处理）。

- 解决数据粒度不匹配的问题（如输出进程每次生成一块数据，而I/O设备每次只能输出一个字符）。

- 提高CPU与I/O设备之间的并行性（CPU可快速将数据放入缓冲区，之后去做其他事情，I/O设备慢慢从缓冲区取数据）。

（二）缓冲区的类型

1. 单缓冲

处理流程：设备→缓冲区→工作区→CPU处理；处理一块数据平均耗时Max(C, T) + M（C为CPU处理时间，T为I/O传输时间，M为数据从缓冲区到工作区的移动时间）；初始状态：工作区满，缓冲区空。

2. 双缓冲

处理一块数据平均耗时Max(T, C+M)；初始状态：工作区空，一个缓冲区满，另一个缓冲区空；两台机器通信时，双缓冲区可实现同一时刻双向数据传输。

3. 循环缓冲

多个缓冲区链接成循环队列，in指针指向第一个空缓冲区，out指针指向第一个满缓冲区，用于批量处理数据。

4. 缓冲池

由系统中共用的缓冲区组成，按使用状况分为：空缓冲队列、装满输入数据的缓冲队列（输入队列）、装满输出数据的缓冲队列（输出队列）。

四种工作缓冲区：用于收容输入数据的工作缓冲区（hin）、用于提取输入数据的工作缓冲区（sin）、用于收容输出数据的工作缓冲区（hout）、用于提取输出数据的工作缓冲区（sout）。

核心流程：①输入进程请求输入数据，通过hin收容输入数据；②计算进程想要取得输入数据，从输入队列中取一块满缓冲区作为sin，读取数据后将缓冲区挂回空缓冲队列；输出流程类似，通过hout收容输出数据，sout提取输出数据。

九、磁盘

（一）磁盘结构

1. 核心概念

- 磁盘：由表面涂有磁性物质的圆形盘片组成，多个盘片“摞”起来。

- 磁道：每个盘片被划分为一个个环形轨道。

- 扇区：每个磁道被划分为一个个扇区，是磁盘读/写的最小单位。

- 盘面：每个盘片有两个盘面（上下表面），每个盘面有一个磁头（移动头磁盘）。

- 柱面：所有盘面中相对位置相同的磁道组成柱面。

2. 磁盘物理地址

格式：（柱面号，盘面号，扇区号）；选择该格式的原因：读取地址连续的磁盘块时，可减少磁头移动消耗的时间（若用“盘面号，柱面号，扇区号”，会增加磁头移动次数）。

3. 磁盘分类

- 按磁头是否可移动：固定头磁盘（每个磁道一个磁头）、移动头磁盘（每个盘面一个磁头）。

- 按盘片是否可更换：固定盘磁盘、可换盘磁盘。

4. 减少延迟时间的方法

- 交替编号：让编号相邻的扇区在物理上不相邻，避免读取完一个扇区后，因处理时间导致错过下一个扇区。

- 错位命名：让相邻盘面的扇区编号“错位”，原理与交替编号一致，降低延迟时间。

（二）磁盘调度算法

磁盘读/写操作所需时间：寻找时间（寻道时间，启动磁臂移动磁头）+ 延迟时间（将目标扇区转到磁头下方）+ 传输时间（读/写数据）；调度算法主要优化寻找时间。

假设场景：磁头初始位置100号磁道，请求访问的磁道号：55、58、39、18、90、160、150、38、184；磁盘磁道范围0~200（部分算法需用到）。

1. 先来先服务（FCFS）

规则：按进程请求访问磁盘的先后顺序调度。

磁头移动路径：100→55→58→39→18→90→160→150→38→184；总移动距离498个磁道，平均寻找长度55.3。

特点：实现简单，无饥饿，但寻道长度长，效率低。

2. 最短寻找时间优先（SSTF）

规则：优先处理与当前磁头位置最近的磁道请求（贪心算法）。

磁头移动路径：100→90→58→55→39→38→18→150→160→184；总移动距离248个磁道，平均寻找长度27.5。

特点：平均寻道时间短，性能较好，但可能产生饥饿现象（如源源不断的近磁道请求，导致远磁道请求无法被处理）。

3. 扫描算法（SCAN，电梯算法）

规则：磁头沿一个方向移动，直到到达最边缘磁道，再反向移动，移动过程中处理沿途的磁道请求。

假设磁头初始向磁道号增大方向移动，移动路径：100→150→160→184→200→58→55→39→38→18；总移动距离282个磁道，平均寻找长度31.3。

特点：无饥饿现象，平均寻道时间较短，但对各个位置磁道的响应频率不平均，且需移动到最边缘才能换向。

4. 循环扫描算法（C-SCAN）

规则：磁头沿一个方向移动，到达最边缘磁道后，直接快速返回起始端（不处理返回途中的请求），再沿原方向处理请求。

假设磁头初始向磁道号增大方向移动，移动路径：100→150→160→184→200→0→18→38→39→55→58→90；总移动距离390个磁道，平均寻找长度43.3。

特点：各个位置磁道的响应频率平均，但平均寻道时间比SCAN长，仍需移动到最边缘换向。

5. 优化算法（低频考点）

- LOOK算法：SCAN的改进，只要在磁头移动方向上不再有请求，就立即改变磁头方向，无需移动到最边缘。

- C-LOOK算法：C-SCAN的改进，只要在磁头移动方向上不再有请求，就立即返回，无需移动到最边缘，返回途中不处理请求。

十、磁盘管理

（一）磁盘初始化

1. Step 1：低级格式化（物理格式化）：将磁盘的各个磁道划分为扇区；一个扇区通常分为头、数据区域（如512B）、尾三部分；管理扇区的数据结构（如校验码）存放在头和尾，用于校验数据是否出错（如奇偶校验、CRC循环冗余校验）。

2. Step 2：磁盘分区：将磁盘分为若干个分区（即C盘、D盘、E盘等），每个分区由若干柱面组成。

3. Step 3：逻辑格式化：创建文件系统，包括创建根目录、初始化存储空间管理的数据结构（如位示图、空闲分区表）。

（二）引导块（启动块）

计算机开机时需执行初始化程序（自举程序），若自举程序放在ROM中，无法更新；解决方案：完整的自举程序放在磁盘的引导块（启动分区）上，ROM中只存放小型“自举装入程序”。

启动流程：开机→运行ROM中的自举装入程序→找到磁盘引导块→将完整自举程序读入内存→完成系统初始化。

十二、网络协议栈实现

（一）核心大纲

- 核心内容：网络处理性能优化、网络协议的分层模型、套接字模型、数据面控制面分离、网络驱动模型。

- 系统流程：Linux系统收包过程（函数视角、数据视角）、Linux系统发包过程。

（二）网络编程模型：套接字编程

1. TCP服务器流程

socket() → bind() → listen() → accept() → recv()/send() → close()

2. TCP客户端流程

socket() → connect() → send()/recv() → close()

3. UDP服务器流程

socket() → bind() → readfrom()/sendto() → close()

4. UDP客户端流程

socket() → sendto()/readfrom() → close()

（三）操作系统架构对比

- 宏内核：应用→系统库→协议栈→网卡；通过系统调用（sys_call）通信。

- 微内核：应用→LibOS→协议栈→网卡；通过进程间通信（ipc）、共享内存通信。

（四）Linux网络数据包接收处理过程（函数视角）

1. 网卡接收数据包，触发以太网中断；

2. 中断响应函数调用netif_rx()，将数据包交给网络软中断（net_rx_action()）处理；

3. 软中断中执行各层处理函数：IP层（ip_rcv()）、TCP层（tcp_rcv()）、UDP层（udp_rcv()）、ICMP层（icmp_rcv()）等，进行解封装（“拨头”）；

4. 处理完成后，调用data_ready()，将数据从内核空间的sk_buff对象拷贝到用户内存；

5. 唤醒拥有该套接字对象的用户态进程，完成数据接收。

（五）网络包的管理

1. 核心要求

- 高效处理分层：发包时逐层添加协议头部，收包时逐层剥去头部。

- 避免连续存放网络包：减少数据拷贝开销。

- 快速申请和释放内存：防止内存碎片。

2. sk_buff（内核网络报文载体）

作用：用于Linux网络子系统各层之间的数据传递，不同协议层通过控制sk_buff共享网络报文。

（1）核心结构

包含next/prev（链表指针）、len（总长度）、data_len（数据长度）、head/data/tail/end（缓冲区指针）等；私有信息（如TCP的Seq编号、ACK编号、重发标记、应答时间戳）存放在cb缓冲区。

（2）关键操作函数

- 分配：struct sk_buff *alloc_skb(int size, gfp_t priority)；GFP_ATOMIC（中断上下文，不可中断）、GFP_KERNEL（可中断，放入等待队列）。

- 浅拷贝：struct sk_buff *skb_clone(struct sk_buff *skb, gfp_t priority)；克隆控制结构，共享报文（用于tcpdump等抓包工具）。

- 深拷贝：struct sk_buff *skb_copy(const struct sk_buff *skb, gfp_t priority)；复制控制结构和报文（用于修改报文，如NAT地址转换）。

- 释放：void kfree_skb(struct sk_buff *skb)。

（3）收包/发包流程

- 收包：网卡→sk_buff（转换以太网帧）→各层剥头→交给用户。

- 发包：创建sk_buff→各层添加协议头部→交给网卡发送。

3. 经典面试题

问题：Linux内核协议栈收包过程中，从网卡到用户进程，网络包的数据至少要拷贝几次？

答案：至少2次；① DMA拷贝（网卡→内核sk_buff）；② 内核态→用户态拷贝（sk_buff→用户内存）。

十三、最新网络协议栈（性能优化）

（一）Linux网络协议栈的问题

网络设备速度不断提升（从100Mbps到1.6T），CPU向多核方向发展，内核协议栈逐渐成为高性能网络的性能瓶颈。

（二）数据面与控制面分离

核心思路：将网络处理分为数据面（快速处理数据包收发）和控制面（管理配置、策略），提升处理效率。

（三）软件优化方案：Intel DPDK

1. 核心特点

- 绕过Linux内核协议栈（bypass kernel），直接在用户空间实现数据包的收发和处理。

- 采用Poll Mode Driver（PMD），不用中断，改为轮询，减少中断开销。

- 支持用户态IO（Linux User I/O），通过mmap()与外设共享内存，实现零拷贝；通过/dev/uioX读取中断（硬中断仍在内核处理）。

2. 核心组件

- 无锁环（Lockless Ring）：解决多线程/多进程并发访问的锁竞争问题，提升并发性能（如多生产者入队用CAS机制，消费者出队移动指针）。

- 内存池（Memory Pool）：快速申请和释放内存，避免内存碎片。

- 扩展框架：支持多种应用场景扩展。

3. DPDK使用特点

- 本身仅支持2层协议，不提供socket接口，适合软路由场景。

- 用于应用程序需提供socket抽象：类微内核方案（DPDK+TCP/IP协议栈作为服务器）、LibOS方案（逐个应用添加协议栈支持）。

- 基于DPDK的完整协议栈框架：腾讯云F-stack、Linux基金会Vector Packet Processor（VPP）、mtcp-stack等。

（四）硬件优化方案：RDMA

1. 核心原理

Remote Direct Memory Access（远程直接内存访问），在远端无感的情况下，直接读取远端内存数据，完全绕开内核协议栈，减少CPU干预。

2. 与传统网络对比

- 传统网络：应用→TCP/IP协议栈→驱动→网卡→远端网卡→驱动→TCP/IP协议栈→应用（多层转发，CPU干预多）。

- RDMA网络：应用→RNIC（RDMA网卡）→远端RNIC→应用（直接内存访问，绕开协议栈）。
